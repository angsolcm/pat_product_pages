PAT Product Pages

One aspect of PAT's remit is as a central source for product usage stats and trends.

It's important to understand, however, that there are vastly different product usage information requirements across the various functions of Angel. For example:

  • Renewals need to identify and prevent accounts which are waning while also identify and utilise the relationships with strong accounts for referrals. This needs to be supported by a robust and accurate account usage measurement system.
  • In making decisions about resourcing it's important for business analysts to understand the engagement with a specific module or feature.
  • Marketing need to isolate accounts and roles based on complex criteria for communications targeting.
  • In supporting a customer, the support team need to know minutia about an account to help resolve individual issues.

Views

We can cover the vast majority of use cases for product usage information with four interrelated views:

  1. Global
  2. Account List
  3. User Account List
  4. Individual Account

An additional set of Module/Feature views may be necessary for more complex products where module-specific usage is of use.

Below we outline the standard views we want present in PAT for each product, it's rationale and use cases. The accompanying images are of current examples for Perspective, but don't necessarily reflect final designs.

Global

The umbrella view of the product and springboard to more specific analysis and lists.

The page is organised into a series of dynamic panels, each with a specific focus. Each panel is completely customisable depending on need.

While the visible panels will be product specific we can minimally expect there to be panels about the following topics:

  • Accounts
  • Users
  • Modules/Features
  • Licences/Opportunities

Certain values are links which navigate the user to an automatically filtered version of the Account List or User Account List pages.

Account List

A filterable and sortable list of all product accounts detailing account features and usage.

This page also acts as a springboard to the Individual Account page for a clicked school.

Filtering and sorting on this page would enable someone to isolate groups of schools that meet certain criteria; for mailing list purposes, for example.

There is also the potential to show summary usage information about the filtered schools.

Since the amount of information we can want to know about accounts can be many and varied, paired with the limited amount of horizontal space, we are currently prototyping how we can show dynamic panels of information within the table.

The technical challenge with these pages is engineering the ever-expanding list of filters driven by the query string.

User Account List

In more common parlance, a User Account List page is a "User Finder".

On these filterable and sortable pages we describe the features and usage of the individual user account, where appropriate.

While historically these pages have been used to find user credentials we now need to display further information regarding role, permissions, activity, contact info, and more.

Again, because these tables can become cluttered with trying to show everything about a user's account, we will likely switch to a dynamic, panelled view.

Similar to the Account List page, the technical challenge is engineering an extensible filtration system using the passed in query string parameters.

Individual Account

The individual account page is the lowest level generic view we would offer as standard for a product, although for some products it might be useful to supply an individual user account page.

This page is a particular use to those functions that have contact with individual accounts - namely support and renewals.

They help identify year-on-year usage changes, discrepancies in implementation, module-specific interactions and engagement, licence information and history, weekly activity information.

Via query string parameters we'd also make these page visible at appropriate locations within ACT, therefore building once but utilising in multiple locations.

It's important to understand we'd make a definitive distinction between showing account product usage and administrating an account. The former would be done on this page and the latter on a separate page or pages.
